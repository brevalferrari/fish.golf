The Hidden songs of Computers

That's a hell of a title isn't it. Allow me to explain.

Sometimes we love music because of an alien logic that is pleasing to try and understand one's own way ; twelve-tone serialism is the perfect example. I believe every song has some kind of structure or logic to it. If you reverse this theory, every audio form with some kind of structure or logic to it can be considered music and can even be pleasing to try and understand.

Computers are full of logic. So they're full of music. All they need is to be able to sing.

Yes, I like Frankenstein. I like to play in the uncanny valley and I find this morbid curiosity fascinating. So as you may know already, computers communicate! They send themselves packets of information over the wire. Just like humans do over the air. So what would it be like to hear them?

Turns out, it was pretty easy to do as you can guess from the relatively small size of the script I came up with. If you have Python and Wireshark / libpcap support via Npcap, it might just work on your computer. I'll try to upload a video when I get it working on a Windows machine as I could not make it work on Linux as of writing this paragraph.

I also made the same thing about hearing keyboard input this time. Spybeep turns your keyboard into a synth keyboard but keeps its functionality. Do not type passwords with it! If you have Rust, you can install it with "cargo install spybeep".

Talking about turning events into sound, why not turn any data into sound as well? I made a tool called playthis which will play anything you have to offer as raw audio. This process is called Databending. I'm currently working on Bent, a project aiming to build a user-friendly tool to bend anything into anything.

Meanwhile, I fiddled with SVG paths, a system that builds SVG elements out of a small language. I did an artistic toy named GraphExpr that builds parametric graphs. You can visit the online version here.

I also have a dead project called Gyromin which is supposed to be an Android app that turns your phone into a Theremin powered by the phone's gyroscope. Android development is a pain but maybe one day I'll get back to it.

Shaping the future by designing new composition tools

Computer assisted music production started from the 50's to the 70's and got popular when computers became cheaper and more powerful. A lot of new tools gave us new ways of making music. As any tool, they also shaped the art we made with it in many ways. For example, most Digital Audio Workstations (DAWs) use "patterns" and allow to easily compose a song with drag-and-drop loops. This made our music easily more repetitive and contributed to completely new genres. A lot of electronic dance genres would probably not exist without this technique which was inspired by common artistic repetition techniques used across all modern history.

Today, the diversity and availability of these tools seems more than ever linked to the diversity of our art styles because we can achieve so much more with a computer that we rely a lot on these tools. So by contributing to the world of computer assisted music production tools, we directly contribute to our artistic diversity.

My first steps into this rabbit hole started when I found BeepComp, an editor for a custom MML variant. I became fascinated by the software but even more by this new way of thinking about music. Languages all have their own logic and constraints. MIDI for example comes with a language that is very precise but almost impossible to write by hand (it's made of bytes) and the standard implies a specific bank of pre-generated sounds and natively only works with a standard set of 12 notes.

When I started making languages for this purpose, I had so many ideas!

The first thing I wanted to try out is to remove the most constraints I could while still keeping my languages simple. I called my first project Blip ; I already could have variably-sized octaves with equal temperaments and when I heard it sounds weird and discovered the difference with the usual equal temperament we use today, it got me interested in xenharmonics. Meanwhile, I was thinking about how language layout impacts its identity and its usage (research repository).

I first came up with LOV (Lengh, Octave, Velocity), an editor for an MML-like language. You can download a zipped EXE here. After that, I decided that MIDI was too restrictive and implemented a variably-sized octave system in Rust and made another language and interpreter, Bleeperpreter.

Bleeperpreter introduced a new way of using instruments : instead of choosing from a list or having to bring a soundfount or samples on the table, you can now make your own instrument. BeepComp used sine, square and triangle waves but you can write them because these are signals with an underlying mathematical expression. Instead of hard-coding expressions, you write them yourself.

Bleeperpreter also had some design flaws. Elapsed time could not be kept between notes and because time is part of any signal expression, there was a slight crack between every note. Also, it did not solve an issue that was already present in BeepComp : how do you deal with crescendos or rituendos? There is no "slopes" to linearly change these parametters.

So BLIP was (re)born. I made yet another language that's completely agnostic on music theory standards. Although there is an equivalent to notes (to which you give your own note names), you decide if octaves are real by binding a variable for it while calling the interpreter and using this variable in the instrument signal expression. Time also has standards in music theory (from note lengths to tempi) and because of that the interpreter needs another expression for note lengths. To solve the "slope" issue, the language allows you to smoothly change variables values by mutating them each frame / sample. These variables can then be bound to anything : note frequencies, note lengths, signal amplitude (volume)...

Thus, I know have a simple language with very little constrains. I still need to do some documentation and tutorials and I'm sure toying with it will bring me more ideas. I hope one day my languages will help other musicians and maybe bring new colors to the world of computer music.
